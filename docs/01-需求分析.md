# 通用账户系统 - 需求分析文档

## 1. 项目概述

### 1.1 项目背景
构建一个高性能、可扩展的通用账户系统，为内部和外部业务系统提供统一的认证授权服务。系统需要支持多租户接入、细粒度的权限管理，并保证高可用性和安全性。

### 1.2 目标用户
- **内部业务系统**: 公司内部各业务线系统
- **外部合作伙伴**: 第三方 API 调用方
- **终端用户**: 最终使用业务系统的用户

## 2. 功能性需求

### 2.1 用户管理模块

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 用户注册 | 支持邮箱/手机号注册，自动发送验证码 | P0 |
| 用户登录 | JWT Token 认证，支持记住登录状态 | P0 |
| 用户信息管理 | CRUD 操作，支持批量导入导出 | P0 |
| 密码管理 | 加密存储、定期强制修改、找回密码 | P0 |
| 多因素认证 | TOTP/短信验证码 (可选启用) | P1 |
| 账号锁定/解锁 | 连续失败自动锁定，管理员手动解锁 | P1 |
| 登录历史 | 记录登录时间、IP、设备信息 | P2 |

### 2.2 组织与部门管理

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 组织架构 | 树形结构，支持无限层级 | P0 |
| 部门管理 | 部门的增删改查，支持合并/拆分 | P0 |
| 用户归属 | 支持主部门和兼任部门 | P0 |
| 岗位管理 | 定义组织内具体岗位 | P1 |

### 2.3 角色管理 (RBAC)

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 角色定义 | 角色名称、编码、描述、数据范围 | P0 |
| 角色继承 | 支持角色层级和继承关系 | P1 |
| 角色分配 | 给用户分配角色，支持批量操作 | P0 |
| 角色复制 | 基于现有角色快速创建 | P2 |

### 2.4 权限管理

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 菜单权限 | 控制用户可见的菜单项 | P0 |
| 按钮权限 | 控制页面内操作按钮的显示 | P0 |
| 数据权限 | 基于部门/个人的数据过滤规则 | P0 |
| 字段权限 | 控制敏感字段的可见性 (如手机号脱敏) | P1 |
| API 权限 | 控制接口访问权限 | P0 |
| 权限继承 | 子节点自动继承父节点权限 | P1 |

### 2.5 认证与授权

| 功能 | 描述 | 优先级 |
|------|------|--------|
| JWT Token 生成 | 包含用户 ID、角色、权限、过期时间 | P0 |
| Token 刷新 | 支持 Refresh Token 机制 | P0 |
| Token 黑名单 | 支持主动失效 (登出/封禁) | P0 |
| 会话管理 | 查看在线用户、强制下线 | P1 |
| 单点登录 | 支持单设备/多设备登录策略 | P1 |

### 2.6 审计与日志

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 登录日志 | 记录所有登录行为 | P0 |
| 操作日志 | 记录关键业务操作 | P0 |
| 审计报表 | 可视化展示审计数据 | P1 |
| 异常告警 | 可疑行为实时告警 | P1 |

### 2.7 开放 API

| 功能 | 描述 | 优先级 |
|------|------|--------|

## 3. 非功能性需求

### 3.1 性能要求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 并发用户数 | 50 万 + | 注册用户总量 |
| 日活用户 (DAU) | 10 万 + | 每日活跃用户 |
| QPS | 1000+ | 峰值每秒查询率 |
| 登录响应时间 | < 200ms | P99 延迟 |
| 鉴权响应时间 | < 50ms | Token 验证延迟 |
| 系统可用性 | 99.9% | 年度 SLA |

### 3.2 安全要求

| 要求 | 实现方式 |
|------|----------|
| 密码存储 | BCrypt 加密，cost factor ≥ 12 |
| 传输加密 | HTTPS/TLS 1.3 |
| Token 安全 | HS512/RS256 签名，有效期可控 |
| 防暴力破解 | 连续失败 5 次锁定 30 分钟 |
| SQL 注入防护 | 参数化查询，ORM 框架 |
| XSS 防护 | 输入过滤，输出转义 |
| CSRF 防护 | Token 验证 |
| 敏感数据脱敏 | 手机号、身份证等字段加密存储 |

### 3.3 可扩展性要求

- **水平扩展**: 支持无状态部署，通过负载均衡扩容
- **数据库分片**: 支持按租户/用户 ID 分库分表
- **缓存层**: Redis 集群，支持热点数据缓存
- **消息队列**: Kafka/RocketMQ异步解耦

### 3.4 可维护性要求

- **健康检查**: 提供/actuator/health端点
- **链路追踪**: 集成Sleuth+Zipkin/Jaeger
- **指标监控**: 集成Prometheus+Grafana
- **日志收集**: ELK/Loki集中日志

## 4. 数据权限模型详解

### 4.1 数据范围类型

```
┌─────────────────────────────────────────────────────────┐
│  数据范围级别                                            │
├─────────────────────────────────────────────────────────┤
│  1. ALL         - 全部数据权限                          │
│  2. DEPT        - 本部门数据权限                        │
│  3. DEPT_AND_CHILD - 本部门及子部门数据权限             │
│  4. SELF        - 仅本人数据权限                        │
│  5. CUSTOM      - 自定义数据范围                        │
└─────────────────────────────────────────────────────────┘
```

### 4.2 权限判断流程

```
用户请求 → 解析 Token → 获取用户角色 → 查询数据范围 → 拼接过滤条件 → 执行查询
```

### 4.3 示例场景

| 角色 | 数据范围 | 可见数据 |
|------|----------|----------|
| 超级管理员 | ALL | 全公司所有数据 |
| 部门经理 | DEPT_AND_CHILD | 本部门及下属部门数据 |
| 普通员工 | SELF | 仅自己创建的数据 |

## 5. API 设计规范

### 5.1 RESTful 风格

```
GET    /api/v1/users          # 获取用户列表
POST   /api/v1/users          # 创建用户
GET    /api/v1/users/{id}     # 获取用户详情
PUT    /api/v1/users/{id}     # 更新用户
DELETE /api/v1/users/{id}     # 删除用户
```

### 5.2 统一响应格式

```json
{
  "code": 200,
  "message": "success",
  "data": { ... },
  "timestamp": 1714291200000
}
```

### 5.3 错误码规范

| 错误码范围 | 说明 |
|------------|------|
| 200 | 成功 |
| 400-499 | 客户端错误 |
| 500-599 | 服务端错误 |
| 1000-1999 | 认证相关错误 |
| 2000-2999 | 权限相关错误 |
| 3000-3999 | 业务相关错误 |
